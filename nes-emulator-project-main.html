RetroDS is a battery-powered handheld NES emulator I designed and built from scratch on an ESP32-WROOM-32E. The firmware runs a C++ NES engine with a full 6502 CPU emulator, PPU graphics renderer, and APU audio synthesiser. Game output goes to a 240×240 IPS TFT display, with stereo audio through a PCM5102A I2S DAC. ROMs are loaded from a MicroSD card at runtime.

The build progressed from a breadboard prototype to a hand-soldered perfboard console. The front board carries the ST7789 display, D-pad, and action buttons; the back carries the ESP32, audio DAC, MicroSD breakout, and power management ICs. A custom 2-layer KiCAD PCB is currently in progress to consolidate everything onto a single compact board.

Hardware Architecture

Five subsystems are interconnected on perfboard via point-to-point soldered wiring. The hardware SPI bus is reserved exclusively for the display — CS is tied permanently low, so no other device can share it. The SD card uses a separate set of GPIO pins running software SPI. All subsystems were validated on breadboard before final perfboard assembly.

04

Firmware

Written in C++ using the Arduino-ESP32 framework. The NES emulation engine handles CPU, graphics, and audio synthesis in a coordinated loop. The I2S audio DMA task runs at elevated FreeRTOS priority to prevent dropouts, while a dedicated video task on Core 0 handles frame conversion and SPI push.

NES Emulation Core
Full 6502 CPU + PPU graphics with cycle-accurate timing
Implements a complete 6502 CPU emulator with cycle-accurate timing and interrupt handling (NMI, IRQ). The PPU renders background tiles and sprites to a 256×240 framebuffer using the NES 64-colour palette. ROM data is loaded from MicroSD into SPI Flash at address 0x00300000 and memory-mapped via spi_flash_mmap() — the emulator reads ROM as a const uint8_t* pointer without consuming DRAM.
Display Subsystem
ST7789 IPS over hardware SPI with double-buffer frame push
The ST7789 display is driven at 27 MHz SPI using the TFT_eSPI library. A video task running on Core 0 converts the PPU’s 8-bit palette indices to 16-bit RGB565 colour, then pushes each frame via tft.pushImage(). A FreeRTOS queue syncs frames between the emulation loop and the display task. tft.setSwapBytes(true) corrects the byte-order expected by the ST7789 panel. Backlight brightness is controlled via the LEDC PWM peripheral.
Audio Subsystem
PCM5102A I2S DAC — 16-bit stereo at 24 kHz with DMA ring buffer
The PCM5102A outputs 16-bit stereo audio at 24 kHz (DEFAULT_SAMPLERATE 24000). The NES APU synthesises square, triangle, noise, and DMC channels and writes samples into an I2S DMA ring buffer (6 buffers × DEFAULT_FRAGSIZE samples). The audio task runs at higher FreeRTOS priority than the PPU loop to ensure the buffer never runs dry under heavy emulation load.
Input Handling
8 GPIO buttons polled per frame with software debouncing
All eight GPIO inputs use internal pull-up resistors. Buttons are polled once per emulation frame (~20 ms at 50 Hz PAL). Debouncing is applied by requiring consistent reads across consecutive frames. The result maps directly to the standard NES controller byte format: A, B, SELECT, START, UP, DOWN, LEFT, RIGHT (bits 0–7).
Cartridge Mapper Support
Mappers 0–5 plus VRC variants — covers the vast majority of NES titles
The mapper number is read from the iNES ROM header and the correct interface is loaded dynamically via mmc_create(). Supported: Mapper 0 (NROM — Donkey Kong, Super Mario Bros.), Mapper 1 (MMC1 — Mega Man 2, Metroid, Zelda), Mapper 2 (UNROM), Mapper 3 (CNROM), Mapper 4 (MMC3), Mapper 5 (MMC5), plus several VRC variants. Maximum ROM size: 512 kB via SPI flash memory mapping.

Display Library Setup

The TFT_eSPI library does not auto-detect which display is connected. Before the firmware will compile correctly, you need to edit a configuration file inside the Arduino libraries folder on your machine — this is not part of the project code. Without this step, the library targets the wrong driver and the screen shows nothing or corrupted output.

Why This Step Is Critical
TFT_eSPI ships with a User_Setup.h file that determines which display driver is compiled, along with the SPI pin assignments and screen resolution. The default targets an ILI9341 240×320 display — not the ST7789 240×240 used here. Flashing without editing this file results in a blank or garbled screen. The change lives in your local Arduino installation, not inside the project sketch folder.
How to Apply the Change
Open the TFT_eSPI configuration file in your Arduino libraries folder:

Arduino\libraries\TFT_eSPI\User_Setup.h

Comment out all existing content in the file, then paste in the configuration shown below and save. The next compile will target the ST7789 at 240×240 with the correct pins.

Note on TFT_CS –1: setting CS to –1 means chip select is permanently active — the display is the only device on that SPI bus, so no switching is needed. The SPI frequency is set to 27 MHz, which is reliable for ESP32 with short perfboard wiring. 
// TFT_eSPI — User_Setup.h
// File location: Arduino\libraries\TFT_eSPI\User_Setup.h
// Comment out ALL existing content in this file, then add:

#define ST7789_DRIVER        // Select ST7789 display driver

#define TFT_WIDTH   240      // Display width in pixels
#define TFT_HEIGHT  240      // Display height in pixels

#define TFT_MOSI   23       // SPI Data
#define TFT_SCLK   18       // SPI Clock
#define TFT_DC      5       // Data / Command select
#define TFT_RST    19       // Reset
#define TFT_CS     -1       // CS always active (display-only SPI bus)

#define SPI_FREQUENCY  27000000   // 27 MHz — reliable for perfboard wiring

#define LOAD_GLCD    // Font 1
#define LOAD_FONT2   // Font 2 (small)
#define LOAD_FONT4   // Font 4 (medium)

Build Progression

Each build phase was fully tested before moving to the next — starting with individual component validation, then breadboard bring-up, perfboard assembly, and finally power integration. This approach caught issues early and kept rework to a minimum.

Phase 1 — Component Sourcing
ST7789 display, PCM5102A audio module, MicroSD breakout, tactile buttons
Sourced all modules independently: ST7789 1.3″ IPS TFT, PCM5102A I2S DAC breakout, MicroSD breakout, 8× tactile push buttons, IP5306 power bank IC, TP4056 USB-C charge module, and a LiPo cell. Each component was bench-tested for correct voltage and pin continuity before integration.
Phase 2 — Breadboard Prototype
First boot — display active, SD card detected, all subsystems validated
Built the first prototype on breadboard with all five subsystems. Achieved first boot with the display showing test output and the SD card detected at 30 GB. Audio confirmed at 24 kHz via PCM5102A. Button GPIO reads verified via serial monitor. The SPI bus conflict between display and SD was identified and resolved at this stage by moving SD to software SPI.
Phase 3 — Perfboard Assembly & Soldering
Hand-soldered front and back boards with colour-coded wiring harness
Components transferred from breadboard to perfboard using a temperature-controlled soldering station with paste flux. The front panel board carries the ST7789 display, D-pad, and A/B/Start/Select buttons. The back board carries the ESP32, PCM5102A, MicroSD breakout, and IP5306 power IC. A colour-coded wiring harness connects the two boards for easy debugging.
Phase 4 — Power Integration & Final Testing
LiPo cell integrated with USB-C charge, 5 V boost, and 3.3 V logic rail
Integrated the LiPo 3.7 V cell with TP4056 USB-C charge IC and IP5306 5 V boost converter. Logic regulated to 3.3 V across all digital subsystems. Full NES ROM loading from MicroSD confirmed working. Audio playback and display rendering verified across multiple games. With all subsystems running reliably, the prototype was complete.

Engineering Challenges

SPI Bus Conflict — Display vs MicroSD
Both the ST7789 display and MicroSD module use SPI. With TFT_CS set to –1, the display holds the hardware SPI bus permanently. Moving the SD card to software SPI (bit-banged) on dedicated pins (MOSI 17 · MISO 13 · SCK 14) freed the hardware bus exclusively for the display at 27 MHz, with no arbitration overhead or frame rate penalty.
I2S Audio Dropouts Under Heavy CPU Load
Initial testing showed audio glitches when the 6502 emulation loop was under heavy load. Fixed by pre-filling the I2S DMA ring buffer at startup and assigning the audio task a higher FreeRTOS priority than the PPU rendering task. This ensures the DMA buffer never runs dry regardless of emulation CPU demand.
GPIO Input Noise on Floating Button Lines
Floating GPIO inputs caused spurious button presses during testing. Fixed by enabling internal GPIO pull-up resistors and debouncing button reads across consecutive emulation frames — no external hardware required, and no interrupt-driven edge detection needed.
ROM Size Exceeds Available DRAM
ROMs larger than available DRAM cannot be copied into RAM. Solved using the spi_flash_mmap API to memory-map ROM data directly from SPI flash starting at address 0x00300000. The emulator accesses ROM as a const uint8_t* pointer without occupying any DRAM, enabling support for ROMs up to 512 kB.

PCB & Next Steps

With the perfboard prototype validated, I am designing a custom 2-layer PCB in KiCAD to replace it. The schematic and layout carry over every verified connection, with key improvements: a dedicated ground plane, separated analogue and digital ground zones for cleaner audio, and a single compact board replacing the two-board perfboard stack.

Custom PCB — KiCAD · In Progress
A 2-layer board with a continuous copper pour ground reference and dedicated 3.3 V and 5 V power polygons. Display and button footprints face the top; ESP32 and audio module on the back. Consolidates the two-board perfboard stack into a single rigid PCB — eliminating all point-to-point wiring and improving mechanical reliability.
3D-Printed Enclosure — Fusion 360
A snap-fit shell designed in Fusion 360 around the final PCB footprint. Features a screen cutout, button holes aligned to D-pad and action buttons, USB-C port access, and a speaker outlet. Printed in PLA on a Creality Ender-3 S1 Pro.
Speaker & Hardware Volume Control
Add a 0.5 W 8Ω speaker for built-in audio and a hardware volume potentiometer on an analog GPIO pin. The PCM5102A DAC supports hardware volume attenuation via its control lines, with software volume as a secondary option.
ROM Selection Menu UI
Build an on-device ROM browser rendered on the ST7789 display — reads available .nes files from MicroSD and shows a scrollable list. D-pad to navigate, A to launch. Replaces the current single hardcoded ROM filename in firmware.

ROM Loading — SPI Flash Memory Mapping

ROM is loaded from MicroSD into SPI flash at 0x00300000, then memory-mapped. The emulator accesses ROM data as a normal pointer without using any DRAM.

#define ROM_FLASH_ADDR   0x00300000
#define ROM_SECTOR_SIZE  0x1000     // 4 kB sectors

// After writing ROM bytes to flash:
const void *romPtr = nullptr;
spi_flash_mmap_handle_t mmapHandle;

esp_err_t err = spi_flash_mmap(
  ROM_FLASH_ADDR,
  romSizeAligned,
  SPI_FLASH_MMAP_DATA,
  &romPtr,

I2S Audio Configuration

I2S driver config for the PCM5102A DAC. 16-bit stereo at 24 kHz, master TX mode. DMA ring buffer ensures smooth audio independent of CPU load.

#define DEFAULT_SAMPLERATE  24000
#define DEFAULT_FRAGSIZE    240   // Samples per DMA buffer

i2s_config_t i2s_config = {
  .mode             = I2S_MODE_MASTER | I2S_MODE_TX,
  .sample_rate      = DEFAULT_SAMPLERATE,
  .bits_per_sample  = I2S_BITS_PER_SAMPLE_16BIT,
  .channel_format   = I2S_CHANNEL_FMT_RIGHT_LEFT,
  .communication_format = I2S_COMM_FORMAT_PCM,
  .dma_buf_count    = 6,
  .dma_buf_len      = 4 * DEFAULT_FRAGSIZE,
  .use_apll         = false,
  .intr_alloc_flags = ESP_INTR_FLAG_LEVEL1
};

i2s_pin_config_t pin_config = {
  .bck_io_num    = I2S_BCK_IO,   // GPIO 27
  .ws_io_num     = I2S_WS_IO,    // GPIO 32
  .data_out_num  = I2S_DO_IO,    // GPIO 25
  .data_in_num   = I2S_PIN_NO_CHANGE
};
